<objective>
Replace the custom language-selector landing UI at `/` with the actual homepage rendered using the default locale, mirroring the arkstudios-website pattern. The root page must include the same nav + footer chrome and the same hero/features/downloaders/FAQ body that `/{locale}/` renders, while keeping `/{locale}/` unchanged for non-default locales.

Purpose: Visitors landing on bitremote.app see the real product immediately instead of a locale picker — better SEO surface, better first impression, fewer clicks.

Output: Two new shared components (`SiteChrome`, `HomeView`), an updated locale layout/page that delegate to them, and a rewritten root `src/app/page.tsx` that composes both with `defaultLocale`.
</objective>

<reference_pattern>
The arkstudios-website project uses this exact pattern (already proven in production):

```tsx
// src/app/page.tsx (root)
export default function Page() {
  return <HomePage language={i18n.defaultLanguage} />
}
```

Their `HomePage` component is rendered by both the root `/` page and the `/[language]/` page, producing identical chrome+body for the default language at both URLs. We are mirroring this — but bitremote already has a working `[locale]/layout.tsx` that owns the chrome (skip-link + TextTabsNav + footer), so we must extract that chrome into a component the root page can also use, since `src/app/page.tsx` does NOT inherit `[locale]/layout.tsx`.
</reference_pattern>
